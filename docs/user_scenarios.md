# Пользовательские сценарии — Видеохостинг-платформа

## Акторы

| Актор           | Описание                                                                                         |
|-----------------|--------------------------------------------------------------------------------------------------|
| **Гость**       | Неавторизованный посетитель платформы. Может только просматривать контент и регистрироваться.     |
| **Пользователь**| Зарегистрированный и авторизованный пользователь. Может загружать видео, комментировать, оценивать и т.д. |
| **Модератор**   | Пользователь с расширенными правами. Может удалять контент, нарушающий правила платформы.         |
| **Администратор**| Полные права в системе. Управление пользователями, ролями, глобальными настройками платформы.     |
    
---

## Сценарий 1 — Гость: Просмотр видео и регистрация на платформе

**Актор:** Гость  
**Цель:** Найти интересующее видео, просмотреть его и зарегистрироваться для получения дополнительных возможностей.  
**Предусловие:** Гость открыл главную страницу видеохостинга.  
**Постусловие:** Гость зарегистрирован и авторизован как Пользователь.

### Основной поток:

1. Гость открывает главную страницу платформы и видит список рекомендуемых/популярных видео (таблица `videos`).
2. Гость вводит поисковый запрос в строку поиска (поиск по полю `title` таблицы `videos`).
3. Система отображает список видео, соответствующих запросу (с названием, превью `preview_filepath`, названием канала из `channels.name`).
4. Гость выбирает видео из списка и переходит на страницу просмотра.
5. Система отображает видеоплеер (поле `filepath`), описание видео (`description`), название канала и комментарии (таблица `comments`).
6. Гость просматривает видео и читает комментарии.
7. Гость хочет оставить комментарий, но система информирует, что для этого необходима авторизация.
8. Гость нажимает кнопку «Регистрация».
9. Система отображает форму регистрации с полями: имя пользователя (`username`), email (`email`), пароль.
10. Гость заполняет форму и отправляет данные.
11. Система валидирует данные (уникальность `username` и `email`), хеширует пароль (`password_hash`), создаёт запись в таблице `users` с ролью «Пользователь» (`role_id` → `roles`), устанавливает `notifications_enabled = true`, фиксирует `created_at`.
12. Система авторизует пользователя и перенаправляет обратно на страницу видео.

### Альтернативный поток:

- **10а.** Введённый `username` или `email` уже существует — система отображает ошибку и предлагает ввести другие данные.
- **10б.** Пароль не соответствует требованиям безопасности — система отображает ошибку валидации.

## Сценарий 2 — Пользователь: Создание канала, загрузка видео и управление плейлистами

**Актор:** Пользователь  
**Цель:** Создать собственный канал, загрузить видео и организовать его в плейлист.  
**Предусловие:** Пользователь авторизован в системе.  
**Постусловие:** Создан канал с загруженным видео, видео добавлено в плейлист.

### Основной поток:

1. Пользователь переходит в раздел «Мои каналы».
2. Пользователь нажимает «Создать канал».
3. Система отображает форму с полями: название канала (`name`, до 32 символов), описание (`description`).
4. Пользователь заполняет форму и подтверждает создание.
5. Система создаёт запись в таблице `channels` с `user_id` текущего пользователя и текущей датой `created_at`.
6. Пользователь переходит на страницу своего канала и нажимает «Загрузить видео».
7. Система отображает форму загрузки: название (`title`, до 64 символов), описание (`description`), видеофайл (`filepath`), превью (`preview_filepath`).
8. Пользователь заполняет форму, прикрепляет видеофайл и превью, нажимает «Опубликовать».
9. Система сохраняет файлы, создаёт запись в таблице `videos` с `channel_id` канала пользователя и текущей датой `created_at`.
10. Если у канала уже есть подписчики, система обновляет данные в таблице `users_channels_subscriptions`, увеличивая счётчик новых видео для подписок на этот канал.
11. Пользователь переходит в раздел «Плейлисты» на странице своего канала и нажимает «Создать плейлист».
12. Система отображает форму: название плейлиста (`name`, до 32 символов), описание (`description`).
13. Пользователь заполняет форму и создаёт плейлист (запись в таблице `playlists` с `channel_id` его канала).
14. Пользователь открывает плейлист и нажимает «Добавить видео».
15. Система отображает список доступных видео.
16. Пользователь выбирает загруженное ранее видео.
17. Система создаёт запись в таблице `playlists_videos` с `playlist_id`, `video_id`, номером в плейлисте (`number`) и датой добавления (`added_at`).

### Альтернативный поток:

- **4а.** Название канала превышает 32 символа — система отображает ошибку валидации.
- **8а.** Формат видеофайла не поддерживается — система отображает ошибку и предлагает загрузить файл в поддерживаемом формате.

## Сценарий 3 — Пользователь: Просмотр видео, подписка на канал, оценка и комментирование

**Актор:** Пользователь  
**Цель:** Посмотреть видео, подписаться на канал автора, оставить оценку и написать комментарий.  
**Предусловие:** Пользователь авторизован в системе.  
**Постусловие:** Создана запись о просмотре, оформлена подписка, оставлены оценка и комментарий.

### Основной поток:

1. Пользователь находит интересующее видео и открывает его страницу.
2. Система начинает воспроизведение видео и создаёт запись в таблице `viewings` (`user_id`, `video_id`, `watched_at`).
3. Пользователь нажимает кнопку «Подписаться» рядом с названием канала.
4. Система создаёт запись в таблице `users_channels_subscriptions` (`user_id`, `channel_id`, `subscribed_at`).
5. Пользователь ставит оценку «Нравится» (лайк) видео.
6. Система создаёт запись в таблице `users_videos_ratings` (`user_id`, `video_id`, `liked = true`).
7. Пользователь пишет комментарий в текстовом поле под видео и нажимает «Отправить».
8. Система создаёт запись в таблице `comments` (`user_id`, `video_id`, `content`, `created_at`).
9. Пользователь открывает комментарий другого пользователя под этим же видео.
10. Пользователь ставит оценку «Нравится» на выбранный комментарий.
11. Система создаёт запись в таблице `users_comments_ratings` (`user_id`, `comment_id`, `liked = true`).

### Альтернативный поток:

- **5а.** Пользователь уже оценивал это видео — система обновляет существующую запись (меняет `liked` на противоположное значение или удаляет оценку).
- **7а.** Комментарий пустой — система отображает ошибку валидации.
- **3а.** Пользователь уже подписан — кнопка отображается как «Отписаться», при нажатии запись удаляется из `users_channels_subscriptions`.

## Сценарий 4 — Модератор: Модерация контента и комментариев

**Актор:** Модератор  
**Цель:** Проверить видео и комментарии на соответствие правилам платформы и удалить нарушающий контент.  
**Предусловие:** Модератор авторизован в системе и обладает соответствующей ролью в таблице `roles`.  
**Постусловие:** Нарушающий контент удалён.

### Основной поток:

1. Модератор открывает панель модерации со списком видео и комментариев.
2. Модератор выбирает конкретное видео и переходит на страницу его просмотра (таблица `videos` по `id`).
3. Модератор просматривает видео, его описание (`description`) и комментарии (`comments`).
4. Модератор определяет, что видео нарушает правила платформы, и нажимает «Удалить видео».
5. Система удаляет запись из таблицы `videos`, а связанные записи в таблицах `viewings`, `users_videos_ratings`, `playlists_videos` и `comments` удаляются каскадно по внешним ключам.
6. Модератор возвращается к панели модерации и выбирает комментарий для проверки.
7. Модератор просматривает комментарий (`comments.content`) и определяет, что он нарушает правила.
8. Модератор нажимает «Удалить комментарий».
9. Система удаляет запись из таблицы `comments`, а связанные оценки из `users_comments_ratings` удаляются каскадно по внешнему ключу.

### Альтернативный поток:

- **4а.** Модератор определяет, что видео не нарушает правила — модератор возвращается к списку материалов на проверке.
- **8а.** Комментарий не нарушает правила — модератор закрывает карточку проверки без удаления комментария.

## Сценарий 5 — Администратор: Управление пользователями и ролями

**Актор:** Администратор  
**Цель:** Управлять учётными записями пользователей и назначать роли.  
**Предусловие:** Администратор авторизован в системе и обладает соответствующей ролью в таблице `roles`.  
**Постусловие:** Роли пользователей изменены, при необходимости пользователь удалён из системы.

### Основной поток:

1. Администратор открывает панель администрирования.
2. Администратор переходит в раздел «Управление ролями» и видит список всех ролей (таблица `roles`: `id`, `name`).
3. Администратор создаёт новую роль (например, «Премиум-пользователь»), система добавляет запись в таблицу `roles`.
4. Администратор переходит в раздел «Управление пользователями» и видит список всех пользователей (таблица `users`: `id`, `username`, `email`, `role_id`, `created_at`).
5. Администратор использует поиск/фильтрацию по `username` или `email` для нахождения нужного пользователя.
6. Администратор выбирает пользователя и открывает его профиль с детальной информацией:
   - дата регистрации (`created_at`);
   - текущая роль (через `role_id` → `roles.name`);
   - каналы пользователя (таблица `channels`);
   - количество загруженных видео (через `channels` → `videos`);
   - количество комментариев (таблица `comments`).
7. Администратор решает повысить пользователя до модератора: изменяет `role_id` в таблице `users` на ID роли «Модератор».
8. Система обновляет запись пользователя в таблице `users`.
9. Администратор находит пользователя через поиск по `username` или `email`.
10. Администратор просматривает связанные с пользователем данные: каналы (`channels`), видео (`videos`), комментарии (`comments`), просмотры (`viewings`) и оценки.
11. Администратор нажимает «Удалить пользователя».
12. Система удаляет запись пользователя из таблицы `users`, а связанные данные удаляются каскадно через внешние ключи: подписки (`users_channels_subscriptions`), оценки видео (`users_videos_ratings`) и комментариев (`users_comments_ratings`), просмотры (`viewings`), комментарии (`comments`), каналы (`channels`), видео (`videos`), плейлисты (`playlists`) и связи плейлист-видео (`playlists_videos`).

### Альтернативный поток:

- **7а.** Выбранная роль уже назначена пользователю — система информирует, что роль не изменилась.
- **11а.** Администратор отменяет удаление на этапе подтверждения — данные пользователя остаются без изменений.
