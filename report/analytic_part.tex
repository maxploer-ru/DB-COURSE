\section{Анализ предметной области}

Предметной областью данного проекта является видеохостинг-платформа, в которой сочетаются две основные стороны работы системы: просмотр контента и его публикация.
С одной стороны, пользователь пользуется платформой как зритель: ищет и просматривает интересующие видео.
С другой стороны, пользователь может выступать как автор, то есть загружать собственные видео в общий доступ.

С точки зрения структуры данных, основные компоненты предметной области включают:
\begin{itemize}
    \item пользователей, которые взаимодействуют с контентом;
    \item каналы, выступающие в роли средства передачи контента от автора к зрителям;
    \item видео, предоставляемые в результате поиска.
\end{itemize}

Для предметной области также характерны следующие пользовательские взаимодействия:
\begin{itemize}
    \item подписка на канал, которая служит основанием для уведомлений пользователя о новых публикациях (под уведомлением понимается любой способ информирования пользователя);
    \item комментарии пользователей к видео;
    \item оценивание видео и комментариев: положительная реакция <<лайк>> или отрицательная --- <<дизлайк>>;
    \item объединение набора видеозаписей в единую структуру <<плейлист>>.
\end{itemize}

Таким образом, требуется хранить не только сведения о самих видеозаписях, но и связи между пользователями, каналами и действиями, которые выполняются в системе.

\section{Формулировка требований к базе данных и приложению}

Необходимо разработать базу данных, которая будет хранить видеофайлы, метаданные и информацию о следующих взаимодействиях пользователей с платформой:
\begin{itemize}
    \item просмотр, оценивание и комментирование видеозаписей;
    \item оценивание комментариев;
    \item создание канала и публикация собственных видео;
    \item объединение видео своего канала в плейлисты;
    \item отправка и получение уведомлений о новых видео.
\end{itemize}

Требуется разработать приложение, предоставляющее возможность осуществлять вышеупомянутые взаимодействия, выполнять поиск видео и получить список видеозаписей какого-либо канала.

\section{Анализ существующих моделей хранения данных}

База данных представляет собой интегрированное хранилище, которое содержит не только сами данные, но и информацию о связях между этими данными, что позволяет пользователю или прикладной программе манипулировать данными, не вникая в детали их физического размещения на носителе~\cite{lit4}.

Под моделью хранения данных понимается способ физической и логической организации данных, который определяет, где и как данные фиксируются, как они структурированы и какие взаимосвязи между ними могут быть установлены для обеспечения работы бизнеса~\cite{lit4}.
Рассматриваются три основных типа моделей организации данных~\cite{lit4}:
\begin{itemize}
    \item дореляционные модели;
    \item реляционная модель;
    \item постреляционные модели.
\end{itemize}

\subsection{Дореляционные модели}

Выделяют следующие дореляционные модели хранения данных:
\begin{itemize}
    \item модель на основе инвертированных списков;
    \item иерархическую модель;
    \item сетевую модель.
\end{itemize}

\subsubsection*{Модель на основе инвертированных списков}

Ключевая идея модели заключается в изменении ролей записей и атрибутов.
Вместо хранения списка атрибутов для каждой записи, для каждого значения атрибута строится список идентификаторов записей, которые содержат это значение~\cite{lit5}.
Это позволяет находить записи, удовлетворяющие условию, быстрее, чем если бы проверялась каждая запись~\cite{lit5}.

Файл с инвертированными списками обычно существует вместе с основным файлом данных, предоставляя избыточную, дублирующую информацию для ускорения поиска~\cite{lit5}.
Пример организации модели представлен на рисунке~\ref{fig:inverted_lists_model}.

\begin{figure}[H]
    \centering
    \includegraphics[width=1\textwidth]{images/inverted_lists_model}
    \caption{Пример организации модели на основе инвертированных списков}
    \label{fig:inverted_lists_model}
\end{figure}

Помимо дублирования данных, среди ограничений также выделяют сложность поддержки при частых обновлениях файла данных: данная модель не оптимизирована для оперативной синхронизации в реальном времени~\cite{lit5}.

\subsubsection*{Иерархическая модель}

Модель организована в виде набора иерархических структур данных (деревьев), где доступ к данным возможен только через корневой узел иерархии~\cite{lit5}.
В иерархической модели поддерживается только связь типа <<один-ко-многим>> (1:N), причём ни одна дочерняя сущность не может иметь более одной родительской сущности~\cite{lit5}.
Схема иерархической модели представлена на примере на рисунке~\ref{fig:hierarchical_model}.

\begin{figure}[H]
    \centering
    \includegraphics[width=1\textwidth]{images/hierarchical_model}
    \caption{Пример организации данных на основе иерархической модели}
    \label{fig:hierarchical_model}
\end{figure}

Иерархическая модель обеспечивает высокую скорость обработки в порядке обхода дерева, но делает произвольный доступ крайне медленным~\cite{lit5}.
Кроме того, из-за запрета на множественное наследование возникают дублирования данных~\cite{lit5}.

\subsubsection*{Сетевая модель}

Сетевая модель появилась вслед за иерархической с целью устранить недостатки последней.
Она снимает ограничение на множественное наследование, позволяет напрямую реализовывать отношения <<многие-ко-многим>> (M:N)~\cite{lit5}.
Схема сетевой модели представлена на рисунке~\ref{fig:network_model}.

Если связь M:N имеет собственные атрибуты, в такой структуре их негде хранить, причём при добавлении третьей сущности для решения этой проблемы связи могут стать двусмысленными~\cite{lit5}.

\subsection{Реляционная модель}

Реляционная модель представляет собой способ организации данных, при котором вся информация представляется в виде двумерных отношений, то есть таблиц, состоящих из строк и столбцов.
Для строки отношение требует единственного значения в каждой ячейке, отсутствия дубликатов и наличия первичного ключа, который однозначно идентифицирует запись.
Взаимодействие с реляционными базами данных выполняется с помощью стандартного языка обработки данных SQL.~\cite{lit5}
Пример представления информации в реляционной модели представлен на рисунке~\ref{fig:relational_table}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{images/relational_table}
    \caption{Пример таблицы реляционной базы данных}
    \label{fig:relational_table}
\end{figure}

Связи между сущностями в реляционной модели задаются через первичные и внешние ключи, а их корректность контролируется механизмом ссылочной целостности, который требует, чтобы каждое значение внешнего ключа соответствовало существующей записи в связанной таблице.
Это позволяет хранить связанные данные в разных таблицах без потери логики предметной области.
Связь вида M:N в такой модели реализуется через промежуточные таблицы, которые могут хранить собственные атрибуты.~\cite{lit5}

\subsection{Постреляционные модели}

Тенденции в глобализации производства формулируют новые требования к хранилищам данных~\cite{lit7}:
\begin{itemize}
    \item создание моделей данных, не требующих строго фиксированной структуры;
    \item хранение в объекте базы данных не только самих данных, но и способов их обработки;
    \item учёт взрывного роста объёмов хранимых данных.
\end{itemize}

Так, были разработаны следующие основные постреляционные модели хранения данных~\cite{lit7}:
\begin{itemize}
    \item модель key-value;
    \item модель объектного хранилища;
    \item объектно-ориентированная модель;
    \item объектно-реляционная модель.
\end{itemize}

\subsubsection*{Модель key-value}

Ключевая идея key‑value модели заключается в хранении данных в виде пар, состоящих из уникального идентификатора и ассоциированного с ним значения.
Данная форма организации данных аналогична ассоциативным массивам в языках программирования.
Для каждого ключа в хранилище предусмотрено ровно одно значение, а доступ к данным осуществляется путём указания ключа с помощью простых команд.
В отличие от реляционных моделей, key‑value хранилища не поддерживают ни вложенности, ни ссылочной целостности.
Key‑value модели характеризуются полным отсутствием фиксированной схемы: данные могут сохраняться в произвольных форматах без предварительного определения таблиц и атрибутов.~\cite{lit6}

\subsubsection*{Объектно-ориентированная модель}

Объектно-ориентированные базы данных обеспечивают создание долговременно хранящихся объектов и средства для поиска и управления этими объектами.
Единство способа хранения данных и парадигмы программирования позволило осуществлять доступ и обработку объектных баз с помощью расширенных средств из языков программирования.~\cite{lit7}

Объектная специфика данных реализуется следующим образом~\cite{lit7}:
\begin{itemize}
    \item объект имеет уникальный идентификатор и наборы изменяемых свойств;
    \item модель содержит исходный набор встроенных объектных и структурных типов, из которых возможно создавать новые типы и моделировать схемы данных;
    \item модель состоит из наборов объектов, каждый из которых является экземпляром конкретного типа.
\end{itemize}

\subsubsection*{Объектно-реляционная модель}

Ограничение реляционной модели данных только нормализованными таблицами не позволяет в рамках одной таблицы представлять
информационные объекты, имеющие сложную внутреннюю
структуру.
Объектно-реляционная модель расширяет реляционный подход системой новых структурных типов данных.
Среди новых типов основными являются~\cite{lit7}:
\begin{itemize}
    \item вложенные таблицы;
    \item ассоциативные массивы;
    \item массивы переменной длины.
\end{itemize}

\subsubsection*{Объектное хранилище}

В объектном хранилище объект является логической коллекцией байтов переменной длины, которая может хранить целые структуры данных: файлы, записи баз данных, изображения или мультимедийный контент.
Каждый объект в такой модели состоит из трёх компонентов: данных, доступных пользователю атрибутов и метаданных, управляемых самим устройством.
Атрибуты описывают характеристики объекта (размер, время доступа, идентификаторы пользователя и группы).
Устройство объектного хранения самостоятельно управляет размещением данных на физическом носителе, включая распределение свободного пространства.~\cite{lit8}

\subsection{Выбор модели}

Применительно к предметной области видеоплатформы, для хранения метаданных о видео и информации о связях, возникающих в системе, выбрана классическая реляционная модель хранения.

Однако реляционная модель не предназначена для хранения произвольных объектов, таких как видеофайлы.
Для обеспечения хранения видео выбрана постреляционная модель, а именно --- объектное хранилище.


\section*{Выводы по разделу}

В аналитическом разделе была проанализирована предметная область видеохостинга, выделены основные сущности и взаимодействия.
