\section{Анализ предметной области}

Предметной областью данного проекта является видеохостинг-платформа, в которой сочетаются две основные стороны работы системы: просмотр контента и его публикация.
С одной стороны, пользователь пользуется платформой как зритель: ищет и просматривает интересующие видео.
С другой стороны, пользователь может выступать как автор, то есть загружать собственные видео в общий доступ. 

С точки зрения структуры данных, основные компоненты предметной области включают:
\begin{itemize}
	\item пользователей, которые взаимодействуют с контентом;
	\item каналы, выступающие в роли средства передачи контента от автора к зрителям;
	\item видео, предоставляемые в результате поиска.
\end{itemize}

Для предметной области также характерны следующие пользовательские взаимодействия:
\begin{itemize}
	\item подписка на канал, которая служит основанием для уведомлений пользователя о новых публикациях;
	\item комментарии пользователей к видео;
	\item оценивание видео и комментариев: положительная реакция <<лайк>> или отрицательная --- <<дизлайк>>;
	\item объединение набора видеозаписей в единую структуру <<плейлист>>.
\end{itemize}

Таким образом, требуется хранить не только сведения о самих видеозаписях, но и связи между пользователями, каналами и действиями, которые выполняются в системе.

\section{Анализ существующих моделей хранения данных}

База данных представляет собой интегрированное хранилище, которое содержит не только сами данные, но и информацию о связях между этими данными, что позволяет пользователю или прикладной программе манипулировать данными, не вникая в детали их физического размещения на носителе~\cite{lit4}.

Под моделью хранения данных понимается способ физической и логической организации данных, который определяет, где и как данные фиксируются, как они структурированы и какие взаимосвязи между ними могут быть установлены для обеспечения работы бизнеса~\cite{lit4}. Рассматриваются три основных типа моделей организации данных~\cite{lit4}:
\begin{itemize}
	\item дореляционные модели;
	\item реляционная модель;
	\item постреляционные модели.
\end{itemize}

\subsection{Дореляционные модели}

Выделяют следующие дореляционные модели хранения данных:
\begin{itemize}
	\item модель на основе инвертированных списков;
	\item иерархическую модель;
	\item сетевую модель.
\end{itemize}

\subsubsection*{Модель на основе инвертированных списков}

Ключевая идея модели заключается в изменении ролей записей и атрибутов. Вместо хранения списка атрибутов для каждой записи, для каждого значения атрибута строится список идентификаторов записей, которые содержат это значение~\cite{lit5}. Это позволяет находить записи, удовлетворяющие условию, быстрее, чем если бы проверялась каждая запись~\cite{lit5}.

Файл с инвертированными списками обычно существует вместе с основным файлом данных, предоставляя избыточную, дублирующую информацию для ускорения поиска~\cite{lit5}. Пример организации модели представлен на рисунке~\ref{img:inverted_lists_model}.

\begin{figure}[H]
	\centering
	\includegraphics[width=1\textwidth]{images/inverted_lists_model.pdf}
	\caption{Пример организации модели на основе инвертированных списков}

Помимо дублирования данных, среди ограничений также выделяют сложность поддержки при частых обновлениях файла данных: данная модель не оптимизирована для оперативной синхронизации в реальном времени~\cite{lit5}.

\subsubsection*{Иерархическая модель}

Модель организована в виде набора иерархических структур данных (деревьев), где доступ к данным возможен только через корневой узел иерархии~\cite{lit5}. В иерархической модели поддерживается только связь типа <<один-ко-многим>> (1:N), причём ни одна дочерняя сущность не может иметь более одной родительской сущности~\cite{lit5}. Схема иерархической модели представлена на примере на рисунке~\ref{img:hierarchical_model}.

\begin{figure}[H]
	\centering
	\includegraphics[width=1\textwidth]{images/hierarchical_model.pdf}
	\caption{Пример организации данных на основе иерархической модели}
	\label{img:hierarchical_model}
\end{figure}

Иерархическая модель обеспечивает высокую скорость обработки в порядке обхода дерева, но делает произвольный доступ крайне медленным~\cite{lit5}. Кроме того, из-за запрета на множественное наследование возникают дублирования данных~\cite{lit5}.

\subsubsection*{Сетевая модель}

Сетевая модель появилась вслед за иерархической с целью устранить недостатки последней. Она снимает ограничение на множественное наследование, позволяет напрямую реализовывать отношения <<многие-ко-многим>> (M:N)~\cite{lit5}. Схема сетевой модели представлена на рисунке~\ref{img:network_model}.
